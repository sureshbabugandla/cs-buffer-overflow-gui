Server-Side Buffer Overflow — Demonstration & Defence

Native C backend (run as an isolated subprocess) · exploitation impact vs. prevention · localhost lab
Educational/defensive lab. It shows that an overflow crashes the server process (availability impact) and corrupts adjacent memory (integrity impact), and how bounds checking, the stack canary / FORTIFY_SOURCE, and memory-safe design prevent it. It contains no control-flow-hijacking exploit, shellcode, or protection bypasses.

① Exploitation — Crash (availability)

Unsafe strcpy into a 64-byte stack buffer, no protections. Long input smashes the stack → process dies.

② Exploitation — Corruption (integrity)

Overflow a 16-byte buffer to overwrite an adjacent authorized variable. Use input length 17–24.

③ Prevention — Bounds check

Same logic, but the code validates length and uses a bounded copy (snprintf). Oversized input is refused.

④ Prevention — Compiler hardening

Same unsafe source, compiled with -fstack-protector-all + FORTIFY_SOURCE. The overflow is detected and aborted safely.
